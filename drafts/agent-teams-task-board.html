01 — Kiến trúc

Mô hình Team

Một team GoClaw gồm một lead agent và nhiều member agents. Lead nhận yêu cầu từ người dùng, phân rã thành tasks trên bảng chung, ủy quyền cho từng member. Kết quả được thu thập và gửi lại một lần sau khi tất cả member hoàn thành.

Điểm khác biệt so với delegation thông thường: mỗi ủy quyền phải gắn với một task trên board — không có task ID thì hệ thống từ chối. Điều này đảm bảo trạng thái luôn nhất quán và có thể audit.

01 — Architecture

The Team Model

A GoClaw team has one lead agent and any number of member agents. The lead receives user requests, decomposes them into tasks on a shared board, and delegates each task to a member. Results are collected and announced back to the lead once all members finish.

The key distinction from plain delegation: every delegation must be linked to a board task — delegations without a task ID are rejected. This ensures state is always consistent and auditable.

01 — 架构

团队模型

GoClaw 团队由一个 主控智能体（lead） 和若干 成员智能体（member） 组成。主控接收用户请求，将任务分解至共享看板，并委派给各成员。所有成员完成后，结果汇总回主控。

与普通委派的关键区别：每次委派必须关联看板上的任务——没有任务 ID 的委派会被系统拒绝，确保状态始终一致且可审计。

01 — アーキテクチャ

チームモデル

GoClaw チームは 1 つの リードエージェント と複数の メンバーエージェント で構成されます。リードはユーザーのリクエストを受け取り、共有ボードのタスクに分解して各メンバーに委任します。全メンバーの完了後、結果は一括してリードに通知されます。

通常のデリゲーションとの違い：すべての委任はボードのタスクに紐づけ必須——タスク ID なしの委任は拒否されます。状態の一貫性と監査可能性が保証されます。

04 — Dependencies

Phụ thuộc blocked_by

Task có thể khai báo danh sách task ID mà nó phụ thuộc vào. Khi tất cả blocker hoàn thành (hoặc bị hủy), task phụ thuộc tự động chuyển từ blocked về pending và được dispatch ngay.

04 — Dependencies

blocked_by Dependencies

Tasks declare a list of prerequisite task IDs. When all blockers are completed (or cancelled), the dependent task automatically transitions from blocked to pending and is dispatched immediately.

04 — 依赖关系

blocked_by 依赖

任务可声明依赖的任务 ID 列表。当所有阻塞任务完成（或取消）后，被依赖任务自动从 blocked 转为 pending 并立即被调度。

04 — 依存関係

blocked_by 依存

タスクは前提タスク ID のリストを宣言できます。すべてのブロッカーが完了（またはキャンセル）されると、依存タスクは自動的に blocked から pending に遷移して即ディスパッチされます。

05 — Dispatch

Luồng Dispatch

Khi lead tạo task trong một lượt hội thoại, hệ thống xếp hàng tất cả task lại (PendingTeamDispatch) và dispatch sau khi lượt kết thúc. Cơ chế này tránh race condition khi thiết lập blocked_by.

05 — Dispatch

Dispatch Flow

When the lead creates tasks during a conversation turn, the system queues all tasks (PendingTeamDispatch) and dispatches them after the turn ends. This prevents race conditions when setting up blocked_by relationships.

05 — 调度

调度流程

主控在一轮对话中创建任务时，系统将所有任务排队（PendingTeamDispatch），并在该轮结束后统一调度，避免设置 blocked_by 时的竞态条件。

06 — Blocker Escalation

Tự động Leo thang

Khi member gặp vấn đề không giải quyết được (API down, thiếu tài liệu...), họ post một blocker comment. Hệ thống tự động xử lý và leo thang lên lead.

06 — Blocker Escalation

Automatic Escalation

When a member hits a hard blocker (API down, missing docs...), they post a blocker comment. The system automatically handles failure and escalates to the lead.

06 — 阻塞升级

自动升级

当成员遇到无法解决的问题（API 故障、缺少文档等），发布一条 blocker comment，系统自动处理失败并上报给主控。

06 — ブロッカーエスカレーション

自動エスカレーション

メンバーが解決できない問題（API 障害、ドキュメント不足など）に直面すると、blocker comment を投稿します。システムが自動で失敗を処理してリードにエスカレーションします。

// Blocker comment triggers auto-fail + escalation
func (b *TeamTasksBackend) handleBlockerComment(
    ctx context.Context,
    task *store.TeamTaskData,
    comment string,
) error {
    // 1. Auto-fail task: in_progress → failed
    if err := b.store.FailTask(ctx, task.ID, comment); err != nil {
        return err
    }
    // 2. Cancel member session
    b.scheduler.CancelSession(task.OwnerAgentID)

    // 3. Broadcast EventTeamTaskFailed → dashboard + WS clients
    b.bus.Broadcast(bus.Event{
        Name:    protocol.EventTeamTaskFailed,
        Payload: buildTaskFailedPayload(task, comment),
    })

    // 4. Escalate to lead via system:escalation
    if b.team.Settings.BlockerEscalation.Enabled {
        return b.escalateToLead(ctx, task, comment)
    }
    return nil
}
