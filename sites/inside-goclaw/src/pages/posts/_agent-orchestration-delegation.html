01 — Vấn đề

Một agent không đủ

Khi bạn giao cho một AI agent một tác vụ lớn — "phân tích toàn bộ codebase và viết báo cáo", "nghiên cứu thị trường rồi tạo slide" — agent đó phải làm tuần tự: đọc từng file, ghi chú, tổng hợp, viết. Chậm. Token explode. Context window tràn.

Giải pháp tự nhiên: chia việc cho nhiều agent chạy song song. Nhưng ngay lập tức nảy sinh bốn câu hỏi:

Q1
Ai quyết định?
Chia việc thế nào, cho ai?
Q2
Kết quả về đâu?
Từ các agent con về lúc nào?
Q3
Agent bị kẹt?
Hệ thống phản ứng ra sao?
Q4
Phối hợp ngang?
Không chỉ nhận lệnh từ trên xuống

GoClaw giải quyết bốn câu hỏi này bằng một hệ thống orchestration đầy đủ: hierarchical delegation, parallel dispatch, task dependency graph, inter-agent mailbox, và automatic workload distribution.

02 — Ba chế độ hoạt động

Spawn, Delegate, Team

Mỗi agent trong GoClaw vận hành theo một trong ba chế độ, được hệ thống tự động xác định dựa trên cấu hình database — không cần người dùng chọn.

Mode · Default
ModeSpawn
Tạo bản sao của chính mình (subagent) để xử lý subtask. Không cần phối hợp với agent khác.
Mode · Expert
ModeDelegate
Liên kết đến các agent khác. Dùng tool delegate để giao việc cho agent chuyên biệt. Routing tới expert agents.
Mode · Full
ModeTeam
Agent tham gia team với lead và members. Có task board chung, mailbox, và workspace chia sẻ.
Ưu tiên resolution: Team › Delegate › Spawn

GoClaw tự phát hiện qua database: kiểm tra bảng teams và agent_links, inject đúng tool set và TEAM.md vào system prompt.

TEAM.md — Virtual file

Nguyên tắc thiết kế quan trọng: Members không nhận TEAM.md đầy đủ — chỉ nhận phần "just do the work". Lead mới có đầy đủ orchestration instructions, danh sách teammates, và patterns phối hợp. Lý do: tiết kiệm token cho idle members.

TEAM.md là virtual file — không lưu trên disk, được render động từ database mỗi khi agent được resolve, inject vào system prompt qua <system_context> tags.

04 — Task Board

Shared task tracker

Task board là trung tâm của hệ thống. Lead tạo task, assign cho member, member claim và thực thi — tất cả qua tool team_tasks.

Vòng đời task — 8 trạng thái

TASK LIFECYCLE STATE MACHINE

in_progress
in_review
blocked
completed
cancelled
stale
failed
pending: chờ claim
in_progress: đang thực thi
in_review: chờ approve
completed
blocked

Atomic Claiming

Hai member không thể claim cùng một task. Thực hiện bằng conditional UPDATE ở database level — không cần distributed mutex:

atomic_claim.sql
UPDATE tasks SET status = 'in_progress', owner = $agent WHERE id = $id AND status = 'pending' AND owner IS NULL -- 0 rows affected → đã bị claim trước
Mandatory task tracking: Lead bắt buộc phải tạo task trước khi delegate. Nếu gọi spawn(agent="member") không có task_id → hệ thống từ chối, hướng dẫn dùng team_tasks(action="create").

05 — Parallel Execution

Nhiều member, cùng lúc

Lead tạo nhiều task cùng một lượt — hệ thống dispatch song song, gom kết quả qua BatchQueue[T]:

PARALLEL DISPATCH FLOW
Task #1 → member_A
▸
Member A chạy
Task #2 → member_B
▸
Member B chạy
Task #3 → member_C
▸
Member C chạy
Lead nhận batch result — một lần announce duy nhất. Synthesis và trả lời user một lượt.

Post-turn dispatch

Tasks được tạo trong lượt của lead được queue, dispatch sau khi lead kết thúc lượt. Tránh race condition với blocked_by setup — không bao giờ dispatch task khi dependency graph chưa hoàn chỉnh.

06 — Task Dependencies

blocked_by graph

Tasks có thể khai báo dependency. Khi task A phụ thuộc task B, hệ thống tự động unblock sau khi B complete:

Task B — phân tích data
─────▸
Task A — viết báo cáo
status: blocked
Khi B complete → DispatchUnblockedTasks() chạy tự động → Task A nhận context đầy đủ từ B
Task A — auto-dispatch · Kết quả B đã append vào context

Cascade Cancel

Cascade cancel: Cancelled task (qua reject hoặc cancel action) cũng unblock dependents — tránh deadlock khi một task thất bại. Hệ thống không để dependents stuck mãi vì một task đã cancel.

Dispatch priority: DispatchUnblockedTasks() tìm tasks có blocked_by đã resolved, dispatch task priority cao nhất mỗi owner trước. Member nhận task với full context — bao gồm kết quả của blocking tasks.

07 — Inter-Agent Communication

Mailbox system

Ngoài task board, members có thể nhắn trực tiếp với nhau qua team_message — message routing qua message bus, không cần polling hay webhook.

Action Mô tả Use case
send Gửi tin cho một teammate cụ thể (theo agent key) Point-to-point
broadcast Gửi cho tất cả teammates trừ bản thân Critical alert
read Đọc tin chưa đọc, tự động mark as read Auto mark-read

Use cases thực tế

Member → Lead: "Task partially complete, cần clarification về requirements"
Lead → Member: "Claim task mới trên board"
Member → Member: Cross-coordination khi hai tasks có overlap

08 — Blocker Escalation

Khi member bị kẹt

Member signal bị blocked bằng blocker comment — hệ thống tự escalate đến lead, không cần lead chủ động monitor:

blocker_comment.pseudo
team_tasks( action="comment", task_id="...", text="Cannot find API docs for payment gateway", type="blocker" )
Task auto-fail: in_progress → failed
EventTeamTaskFailed broadcast tới team
Member session cancelled
Lead nhận escalation từ system:escalation: — đầy đủ context, retry instruction, task_id
Circuit breaker: Task auto-fail sau 3 dispatch attempts (maxTaskDispatches). Tránh infinite loop khi agent không thể hoàn thành một task.

09 — Workload Distribution

Automatic, không cần cấu hình thêm

Priority dispatch
Task priority cao hơn được dispatch trước trong mỗi round.
One-at-a-time per owner
Mỗi agent chỉ nhận 1 task mới mỗi round — sau khi complete task hiện tại.
Concurrency limits
Per-link (mặc định 3 concurrent từ lead đến member) và per-agent. Khi vượt → LLM nhận error message hướng dẫn thử agent khác.
Auto-completion trigger
Khi member complete task → DispatchUnblockedTasks() chạy ngay. Không cần lead can thiệp.
⚡
Adaptive throttle: Khi session history vượt 60% context window, scheduler tự reduce concurrency về 1 — tránh context overflow làm hỏng output quality.
